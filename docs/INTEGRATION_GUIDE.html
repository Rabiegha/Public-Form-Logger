1. Architecture

Le Logger est un service indépendant qui reçoit, valide et stocke chaque soumission de formulaire émise par une landing page en parallèle de l'API Attendee principale. Il n'a aucune logique métier : il logge. 

[ Landing page ]
       │
       ├──► POST /api/event/registration   → Attendee (chemin nominal)
       │
       └──► POST /v1/public-form-logs      → Public Form Logger (filet de sécurité)
  • Les deux requêtes partent en parallèle ; aucune n'est bloquante pour l'autre.
  • Un échec côté Logger ne doit jamais empêcher l'inscription effective côté Attendee.
  • Un échec côté Attendee est rattrapé manuellement (V1) ou automatiquement (V2) via les logs du Logger.

Ordre des champs dans form_payload L'admin du Logger affiche les colonnes dans l'ordre d'insertion des clés. Construisez le payload dans l'ordre où vous voulez voir les colonnes apparaître côté admin. En pratique, parcourir les <input> dans l'ordre du DOM suffit.

4. Codes de réponse

CodeSensAction côté front
201 CreatedLog inséré. Afficher le message de succès à l'utilisateur.
200 OKDuplicate (même submission_id). Considérer comme succès (cas du retry).
400 Bad RequestPayload invalide (token mal formé, payload vide, etc.). Erreur définitive. Ne pas retry.
403 ForbiddenOrigine CORS non autorisée. Vérifier que le domaine est listé dans CORS_ORIGINS côté serveur.
429 Too Many RequestsRate limit dépassé. Retry avec backoff (mais probablement bot — review log).
503 Service UnavailableDB down ou redémarrage. Retry obligatoire avec backoff exponentiel.

5. Idempotence (submission_id)

Le champ submission_id permet au Logger de dédupliquer les tentatives. Si le front envoie deux fois la même soumission (retry réseau, double-clic), le Logger renvoie 200 OK au lieu de créer un doublon.

  • Générez un UUID v4 stable avant le premier envoi.
  • Ré-utilisez le même submission_id pour tous les retries de la même soumission.
  • Régénérez un nouvel UUID après un succès, pour la soumission suivante.
// Génération
const submissionId = crypto.randomUUID();

// Réutilisation pour TOUS les retries de cette soumission
await fetch(url, { body: JSON.stringify({ ..., submission_id: submissionId }) });

6. Stratégie de retry

Retry uniquement sur 503 et erreurs réseau (fetch rejected). Backoff exponentiel.

TentativeDélai avant retry
1500 ms
21 s
32 s
échec finalafficher message d'erreur utilisateur

Ne pas retry sur 4xx (erreurs définitives) ni sur 200/201.

7. CORS & domaines autorisés

Le Logger n'accepte de requêtes POST que depuis les domaines listés dans CORS_ORIGINS côté serveur. Si une nouvelle landing page est déployée sur un nouveau domaine, ouvrir un ticket pour l'ajouter à la whitelist CORS.

Domaines autorisés en production (V1) :

  • choyou.fr (incluant tous les sous-domaines)
  • itforbusiness.fr (incluant tous les sous-domaines)

8. Exemple complet

Une démo end-to-end est disponible dans examples/landing-page/index.html du repo. Elle inclut : payload-builder, retry, idempotence, UTM, honeypot anti-spam. À copier/coller comme starter. 

// Snippet minimal
const submissionId = crypto.randomUUID();

async function submitForm(form) {
  const payload = {};
  form.querySelectorAll('input[name]').forEach((input) => {
    if (input.value.trim()) payload[input.name] = input.value.trim();
  });

  const utm = new URLSearchParams(window.location.search);
  const body = {
    public_token: 'YOUR_PUBLIC_TOKEN',
    submission_id: submissionId,
    form_payload: payload,
    landing_page_url: window.location.href,
    utm_source: utm.get('utm_source') || undefined,
    utm_medium: utm.get('utm_medium') || undefined,
    utm_campaign: utm.get('utm_campaign') || undefined,
  };

  const res = await fetch('https://logger.attendee.fr/v1/public-form-logs', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(body),
  });

  if (!res.ok && res.status !== 200) throw new Error('Logger failed: ' + res.status);
}

9. Checklist d'intégration

  1. Récupérer le public_token de l'événement auprès d'Attendee.
  2. Vérifier que le domaine de la landing est dans la whitelist CORS du Logger.
  3. Générer un submission_id unique côté client (UUID v4).
  4. Construire le form_payload dans l'ordre d'affichage des champs.
  5. Envoyer en parallèle à Attendee (chemin principal) et au Logger (filet de sécurité).
  6. Implémenter le retry exponentiel sur 503 / erreurs réseau.
  7. Ajouter un honeypot anti-spam (input invisible — drop si rempli).
  8. Tester un cas de duplicate (double submit) → vérifier que le Logger renvoie 200.
  9. Tester sans réseau → vérifier que l'UI ne se bloque pas.
  10. Vérifier dans l'admin du Logger que la soumission apparaît avec les bonnes colonnes.

10. FAQ

Que se passe-t-il si Attendee est down mais que le Logger a reçu la soumission ?

Le lead est sauvé. En V1, l'admin du Logger permet de retrouver les leads et de relancer manuellement. En V2, un mécanisme de replay automatique synchronisera les leads manquants.

Le Logger renvoie 503 — que faire côté UI ?

Retry 3 fois en backoff. Si tous les retries échouent, afficher un message d'erreur utilisateur ("Réessayez dans quelques instants"). Ne pas perdre la soumission côté UI : on peut éventuellement la garder en localStorage pour un retry manuel ultérieur.

Faut-il valider l'email côté client ?

Oui — le Logger ne fait pas de validation métier (juste structurelle). C'est au front d'imposer ses règles avant l'envoi.

Est-ce que je peux envoyer des champs custom (pas dans l'exemple) ?

Oui. form_payload est un objet libre (max 100 clés, max 32 KB). Les colonnes dans l'admin se génèrent automatiquement à partir des clés payload reçues.

Que faire avec l'erreur 403 CORS ?

Demander à l'équipe back d'ajouter votre domaine à CORS_ORIGINS. Pas de bypass possible côté front.
