Public Form Logger — Roadmap V2

Document de cadrage · à challenger et prioriser avant implémentation

Contexte

La V1 du Public Form Logger est volontairement simple : service indépendant, stockage des soumissions en base, rôle de filet de sécurité (best-effort), admin panel basique. Elle remplit son objectif initial — ne plus perdre de leads quand Attendee est indisponible — mais plusieurs limites apparaissent à l'usage.

Limites actuelles de la V1

  • Pas d'intégration avec Attendee (auth admin séparée)
  • Aucune garantie d'origine des requêtes (risque de spam)
  • Pas de mécanisme de replay automatique des leads non synchronisés
  • Observabilité limitée (pas d'alerting structuré)
  • Protection anti-spam basique (rate limit IP uniquement)
  • Expérience admin minimale (UI fonctionnelle mais sobre)
  • Pas de distinction entre temps client et serveur (pas de receivedAt)
  • Dépendance forte à la disponibilité de la DB (pas de fallback ni retry serveur)
Objectif : faire évoluer le Logger…
↓
de simple backup passif
↓
vers un système actif de sécurisation, récupération et monitoring des leads

Sommaire des chantiers V2

  1. Intégration Attendee — auth unifiée
  2. Signature des requêtes — sécurité avancée
  3. Migration vers une base managée
  4. Replay automatique des leads non synchronisés
  5. Observabilité & alerting
  6. Anti-spam avancé
  7. Amélioration UX admin
  8. Champ receivedAt côté client
  9. Gestion avancée des pannes DB
  10. Open questions
  11. Conclusion

1Intégration Attendee — auth unifiée priorité haute

Problème V1

  • Auth admin séparée d'Attendee
  • Duplication des comptes utilisateurs
  • Mauvaise UX interne (double login)

Solution V2

  • Auth via JWT Attendee (validation côté Logger via clé publique ou shared secret)
  • Permissions fines (ex. public_form_logs.read)
  • Fallback admin local obligatoire si Attendee est down
Contrainte non-négociable Le Logger doit rester pleinement fonctionnel même si Attendee est indisponible.

2Signature des requêtes — sécurité avancée priorité haute

Problème V1

  • Endpoint public exposé sans signature
  • Aucune garantie que les requêtes viennent de nos landing pages
  • Spam possible avec un simple curl

Solution V2

  • Signature HMAC des requêtes (générée backend, jamais front)
  • Champs signés : public_token, submission_id, timestamp
  • Vérification côté Logger avant insertion
  • Bénéfices : protection anti-spam, intégrité, contrôle des sources

3Migration vers une base managée priorité moyenne

Problème V1

  • Backups gérés manuellement
  • Risque de perte de données
  • Maintenance infrastructure récurrente

Solution V2 — candidates

  • Supabase
  • Neon
  • AWS RDS
  • Cloud SQL (GCP)

Bénéfices : backups automatiques, haute disponibilité, charge ops réduite.

4Replay automatique des leads priorité haute

Problème V1

  • Si Attendee échoue, le rattrapage est 100 % manuel
  • Risque d'oubli ou de retard sur des leads chauds

Solution V2

  • Détection automatique des logs non synchronisés avec Attendee
  • Retry périodique (worker / cron)
  • Replay manuel disponible depuis l'admin

Nouveau modèle de statut

received   → soumission insérée dans le Logger
synced     → confirmée comme présente dans Attendee
failed     → Attendee a refusé après N retries (à investiguer)
Promesse V2 Zéro perte de leads — recovery automatisée bout en bout.

5Observabilité & alerting priorité moyenne

Problème V1

  • Visibilité limitée sur le trafic et les erreurs
  • Aucun alerting proactif

Solution V2

  • Alerting si volume de logs anormalement bas (ex. 0 logs / 1h pendant un événement actif)
  • Dashboard métriques (latences, taux d'erreur, throughput)
  • Logs structurés JSON
  • Intégration Sentry, Datadog ou équivalent

6Anti-spam avancé priorité moyenne

Problème V1

  • Rate limit IP simple, contournable
  • Pas de détection comportementale

Solution V2

  • Honeypot fields côté landing (déjà préconisé en V1, à formaliser)
  • Détection de patterns (payloads identiques, fréquence anormale)
  • Blocage IP / pays / ASN
  • Rate limiting dynamique selon score de confiance

7Amélioration UX admin priorité basse

Problème V1

  • UI fonctionnelle mais minimale
  • Pas de filtres avancés ni de recherche full-text

Solution V2

  • Export CSV / Excel ✅ (déjà livré en V1.x)
  • Filtres avancés (date, événement, statut sync, UTM)
  • Recherche full-text dans le payload
  • Dashboard simple (volumes, événements actifs)

8Champ receivedAt côté client priorité basse

Problème V1

  • Impossible de distinguer le temps client du temps serveur
  • Difficile de mesurer la latence réseau / d'analyser des bugs UI

Solution V2

  • createdAt → timestamp serveur (source de vérité)
  • receivedAt → timestamp client (optionnel, debug)

Règles d'usage

  • Champ optionnel — fallback sur createdAt si absent
  • Ne jamais faire confiance à 100 % au temps client (manipulable)
  • createdAt reste la source de vérité légale et métier

9Gestion avancée des pannes DB à explorer

Décision V1 (conservée)

  • Si DB down → renvoyer 503
  • Le frontend retry avec backoff (responsabilité côté UI)

Réflexion V2

  • Option à explorer : fallback stockage temporaire (file / queue locale)
  • Permettrait d'absorber des pannes DB courtes sans perte
Attention — risques Complexité élevée, risque de corruption, nouveau point de défaillance. Décision actuelle : rester sur l'approche simple et investir plutôt dans le replay automatique (§4) et l'observabilité (§5).

?Open questions à trancher

  • Faut-il introduire une queue (Redis / BullMQ) pour le replay et le découplage ?
  • Faut-il un buffer mémoire temporaire pour absorber les micro-pannes DB ?
  • Faut-il un service intermédiaire dédié à la signature des requêtes ?
  • Quel niveau d'intégration Attendee : SSO complet ou simple validation JWT ?
  • Quel curseur sécurité / complexité accepter (trade-off ROI) ?
  • La signature côté backend implique-t-elle un proxy par landing ? Quel impact ops ?

✓Conclusion

La V2 ne doit pas être une simple évolution technique, mais un changement de posture : 

V1 → stockage passif (best-effort)
V2 → système actif, sécurisé, observable et résilient

Ce document est une base de réflexion. Avant toute implémentation, il devra être :

  • Revu avec l'équipe technique et produit
  • Challengé sur chaque chantier (coût vs valeur)
  • Priorisé par incréments livrables (pas de big bang)
